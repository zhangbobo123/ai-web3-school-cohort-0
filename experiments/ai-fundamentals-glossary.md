# AI 基础概念速查

> 用自己的话整理，用于建立后续学习 Agent、Workflow、AI Coding 和工具调用的共同语言。
> 日期：2026-05-20

---

## 1. LLM（大语言模型）

**一句话**：一个读了海量文本后，能根据上下文预测"下一个词该是什么"的统计模型。

**例子**：你在 ChatGPT 里输入"以太坊的共识机制是"，它接出"权益证明（Proof of Stake）"——不是因为"理解"了共识，而是训练数据里这段话后面大概率跟着这个组合。

**误区**：容易把 LLM 当成"搜索引擎"或"知识库"。实际上它会编造不存在的事实（幻觉），因为它做的是概率续写，不是查表。链上合约地址、实时余额这类精确事实，LLM 本身永远给不了，必须靠工具读取。

---

## 2. Prompt（提示词）

**一句话**：你给模型的任务说明书——目标是什么、边界在哪、输出格式怎样。

**例子**："请对比 Uniswap V2 和 V3 的做市机制，用中文，列三点差异，每点不超过 50 字。"对比只说"讲讲 Uniswap"，前者给出的结果聚焦得多。

**误区**：以为 prompt 越长越好。实际上过长会把关键指令淹没在噪声里；更有效的做法是结构化：角色→任务→约束→输出格式。另外 prompt 不是"编程"，它是对模型的引导而非精确指令，同样的 prompt 不同模型甚至同一模型不同版本都可能给出不同结果。

---

## 3. Context Window（上下文窗口）

**一句话**：模型一次能"看见"的文本长度上限，超出就看不到前面的内容。

**例子**：GPT-4 的 128K context window 大约能装下一本小书。如果你让 Agent 分析一份 200 页的白皮书，前 100 页的内容在处理后半段时就"看不见"了，除非用 RAG 或摘要压缩来管理。

**误区**：context window = 记忆容量。不是的，它是单次对话的"视野范围"，关掉对话就清空。而且即使窗口够大，模型对中间部分的注意力也弱于首尾（"中间遗忘"现象）。做链上 Agent 时尤其要注意：不要把关键约束埋在长上下文的中间位置。

---

## 4. Agent（智能体）

**一句话**：一个能反复"思考→调用工具→观察结果→再思考"的循环系统，而不是只给一次回答。

**例子**：你跟普通 ChatGPT 说"帮我查一下 Vitalik 的 ETH 余额"，它会告诉你"我无法访问链上数据"。但一个配了链上查询工具的 Agent 会：① 判断需要调用工具 → ② 调用区块链 API → ③ 拿到余额 → ④ 格式化返回给你。它不只是一个回答者，而是一个执行者。

**误区**：以为"接了工具 = Agent"。加了 API 调用只是第一步，真正的 Agent 还需要状态管理（知道自己做到哪一步了）、错误处理（工具调用失败怎么办）、权限边界（什么操作能自动做，什么必须问人）。没有这些，它只是"能调工具的聊天机器人"，不是可靠的 Agent。

---

## 5. Tool Use（工具调用）

**一句话**：让模型在需要时主动调用外部接口获取数据或执行操作，而不是全靠自己的"记忆"回答。

**例子**：DeFi 产品里的 Agent 需要查实时价格——模型本身不知道当前 ETH/USDC 汇率，但可以调用 DEX API 或链上 RPC 来获取。工具把"我不知道"变成"我去查"。

**误区**：最大的误区是认为模型自己能决定"该不该调工具"。实际上必须用确定性规则（policy）来约束：哪些工具只读、哪些可写、哪些需要模拟后确认、哪些直接拒绝。模型可以做"选择"，但工具侧必须用硬边界兜底，不能让模型自己定义权限范围。

---

## 6. Workflow（工作流）

**一句话**：把一个复杂任务拆成有序的步骤，每一步有明确的输入、输出和状态，而不是让模型一口气自由发挥。

**例子**：用户说"帮我 swap 100 USDC 换 ETH"。如果直接让模型调工具，它可能跳过模拟直接发交易。有 workflow 的 Agent 会按步骤走：① 读余额 → ② 查价格 → ③ 生成交易参数 → ④ 模拟（Tenderly）→ ⑤ 展示滑点和风险 → ⑥ 用户确认 → ⑦ 发送交易 → ⑧ 追踪结果。每一步出错了都知道停在哪、怎么恢复。

**误区**：以为 workflow 是限制 Agent 能力。恰恰相反，workflow 是让 Agent 在真实场景中可用的前提——没有流程控制的 Agent 在 demo 里看起来很聪明，在涉及资产和权限的场景里是定时炸弹。

---

## 7. Guardrails（护栏）

**一句话**：硬编码的安全规则，在模型输出之后、实际执行之前拦截危险操作。

**例子**：用户让 Agent "approve 无限额的 USDT 授权给这个合约"。模型可能照做，但 guardrail 检测到 approve amount = MAX_UINT256，直接拒绝执行并返回"无限授权存在安全风险，已拦截"。

**误区**：以为 guardrails 就是"让模型说 No"。模型说 No 是软性的，可以被 prompt 绕过；guardrail 是代码层面的硬拦截，模型无法绕过。好的 guardrails 设计是在工具执行前做确定性检查，而不是依赖模型自己的判断力。

---

## 8. Human-in-the-loop（人工介入）

**一句话**：在关键风险点让人做最终决策，而不是让 Agent 全自动执行所有步骤。

**例子**：一个链上交易 Agent 可以自动查余额、自动模拟交易、自动生成交易参数，但发送交易这一步必须等用户点"确认"。不是每一步都需要人——查余额可以自动，但动钱不行。

**误区**：要么全自动化要么每步都确认，这是两极思维。实际上 human-in-the-loop 是分层的：只读操作自动执行，小额白名单按 session key 执行，高风险操作必须人工确认，超出 policy 直接拒绝。重点不是"有没有人工确认"，而是人在确认时能否看懂面前的资产变化和风险。

---

## 概念间的关联

```
Prompt → 告诉 LLM 目标和边界
  ↓
LLM → 在 Context Window 内思考
  ↓
Agent → 循环"思考→行动→观察"
  ↓
Tool Use → 调用外部工具获取数据/执行操作
  ↓
Workflow → 把以上组织成有序可控的流程
  ↓
Guardrails → 在执行前拦截危险操作
  ↓
Human-in-the-loop → 关键节点让人决策
```

每个概念单独看都不复杂，但组合起来才是完整系统——这也是 AI × Web3 产品跟普通 AI 产品的区别：链上操作有资产和权限风险，每个环节都不能省。
