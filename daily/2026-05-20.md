## 学习笔记

### Chain-aware Context（链感知上下文）

**核心观点**：链上数据是不断变化的，AI 模型不能靠记忆来判断链上事实，必须通过工具实时读取。

传统 AI 上下文来自文档、聊天记录、数据库，相对稳定。但 Web3 多了一层——链上状态。余额会变、授权会变、合约会升级、价格会波动，同一个地址在两个区块之间可能完全不同。

**为什么模型记忆不可靠？**
模型知道"Uniswap 是 DEX"没问题，但执行交易时需要具体事实：哪条链、哪个合约、哪个池子、当前价格、用户余额、授权额度、滑点、模拟结果。这些全部有时效性，上一秒对的结论下一秒可能过期。

**三个要求**：
1. 带时间性 — 过期的上下文不能当事实，落后 500 区块的索引结果就不是当前状态
2. 带来源 — 给 Etherscan 链接比说"交易成功"可靠，用户可以自己核验
3. 区分事实和解释 — 工具返回事实，模型负责解释，不能把猜测当事实

**产品启发**：这一层是所有链上 Agent 的输入层。值得追问——现有 AI+Web3 产品哪些真正接入了链上数据，哪些只是在"猜"？

> 来源：https://aiweb3.school/zh/handbook/bridge/chain-aware-context/

### Web3 工具调用（Web3 Tool Use）

**核心观点**：把链上操作变成 Agent 可调用的工具，真正难的不是"能调用"，而是权限、参数、模拟和日志。

与 Chain-aware Context 的关系：上一篇解决**输入问题**（让 AI 看到正确的链上数据），这篇解决**执行问题**（让 AI 能调用链上操作，但要有明确边界）。

**三个核心原则**：
1. 读写分离 — 读链状态和发交易必须是不同工具、不同权限
2. 参数结构化 — chain id、合约地址、method、args、value、slippage 不能埋在自然语言里
3. 日志不可省 — 每次调用都记录输入、输出、时间、来源和错误

**关键洞察**：模型可以选择工具，但工具必须用确定性边界限制模型。不能让 Agent 自己决定权限范围。

> 来源：https://aiweb3.school/zh/handbook/bridge/web3-tool-use/

### 智能体工作流（Agent Workflow）

**核心观点**：把"用户目标 → 上下文读取 → 计划生成 → 工具调用 → 风险检查 → 执行 → 记录和复盘"组织成可控流程，而不是让模型自由发挥。

**为什么重要**：链上动作有资产、权限和不可逆结果，Agent 工作流必须比普通聊天更严格。核心是把概率模型放进确定性流程里。

**六个知识节点**：

1. **Task Graph（任务图）** — 把目标拆成节点和依赖，不让 Agent 一口气自由执行。例如 swap 拆成9步：读目标→获取余额→查价格→生成交易→模拟→展示风险→用户确认→发送→追踪。每步都可设输入、输出、权限和停止条件。

2. **State Machine（状态机）** — 让 Agent 有明确状态（draft→context_loaded→plan_ready→waiting_user_confirmation→submitted→confirmed/reverted/cancelled），不是只有聊天历史。刷新页面、交易 pending、RPC 失败时，系统不会忘记自己在哪，不会重复危险动作。

3. **Human-in-the-loop** — 把人放在关键风险点，不是每步都确认。分层：只读→自动 / 草稿→自动生成 / 小额白名单→session key / 高风险→必须人工 / 超出policy→拒绝。重点是人确认时能否看懂资产、权限变化和失败风险。

4. **Retry / Fallback** — 读余额失败可重试；发交易失败先判断是否已广播；交易 pending 不能再发一笔；模型不可用降级只读模式，不能自动换未评估模型继续发交易。

5. **Trace** — 每步输入、判断、工具调用和结果的记录。至少包括：用户目标、模型版本、上下文来源、工具I/O、policy判断、模拟结果、人工确认、交易哈希、最终状态。没有 trace 只能看聊天记录；有 trace 才能复盘。

6. **Evaluation Harness** — 评估框架不仅测回答好不好，还要测是否拒绝越权、是否识别错误链/合约、是否在缺数据时停止。Regression Set 用固定用例防退化（正常swap/错误链/无限approve/恶意诱导等）。

**与前面章节的关系**：Agent Workflow 是桥接层的流程骨架。Chain-aware Context 提供事实，Web3 Tool Use 提供能力，Agent Wallet 提供权限边界，Workflow 把它们组织成可执行但可控的路径。

**产品启发**：没有 workflow 的项目 = "模型直接调用工具"，demo 里很快，真实资产面前不够用。做产品研究时值得追问：现有 AI+Web3 产品哪些有真正的 workflow，哪些只是把模型接了个工具就上线了？

> 来源：https://aiweb3.school/zh/handbook/bridge/agent-workflow/


### AI 基础概念速查（8个核心概念）

整理了 8 个 AI 基础概念，每个包含一句话解释、具体例子和常见误区：

1. **LLM** — 概率续写不是查表，链上精确事实必须靠工具
2. **Prompt** — 结构化比长度重要，prompt 是引导不是编程
3. **Context Window** — 视野范围不是记忆，中间位置容易遗忘
4. **Agent** — 接了工具不等于 Agent，还需要状态管理和权限边界
5. **Tool Use** — 模型做选择，工具侧用硬边界兜底
6. **Workflow** — 不是限制能力，是让 Agent 在真实场景可用
7. **Guardrails** — 代码层硬拦截，不是依赖模型说 No
8. **Human-in-the-loop** — 分层介入，关键是人能否看懂风险
