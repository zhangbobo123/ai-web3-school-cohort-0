# 链感知上下文（Chain-aware Context）学习笔记

## 核心观点

链感知上下文要解决一个根本问题：**链上数据是不断变化的，AI 模型不能靠记忆来判断链上事实，必须通过工具实时读取。**

传统 AI 应用的上下文来自文档、聊天记录、数据库，这些数据相对稳定。但 Web3 多了一层——链上状态。余额会变、授权会变、合约会升级、价格会波动，同一个地址在两个区块之间可能完全不同。如果 AI 只凭"我记得这个地址有多少余额"来操作，结果就是错的，甚至是危险的。

## 为什么模型记忆不可靠？

模型知道"Uniswap 是 DEX"这种知识没问题，但真正执行交易时需要的不是常识，而是具体事实：哪条链、哪个合约地址、哪个池子、当前价格、用户余额、授权额度、滑点容忍度、交易模拟结果。这些信息全部有时效性，上一秒还对的结论下一秒可能就过期了。

所以核心原则是：**链上事实必须从工具和索引层读取，不能靠模型记忆或用户口述。**

## 构建链感知上下文的三个要求

1. **带时间性**：每条数据都要标注读取时的区块号或时间戳，过期的上下文不能当事实用。比如一个落后 500 个区块的索引结果，就不该被当成当前状态。

2. **带来源**：合约地址、交易哈希、区块号、explorer 链接都要能追溯。给用户一个 Etherscan 链接，比说一句"交易成功"可靠得多——用户可以自己打开核验。

3. **区分事实和解释**：工具返回的是事实（余额、交易记录、事件日志），模型负责解释（这是什么意思、该怎么做）。不能把模型的猜测当成链上事实。

## 七个知识节点速览

- **On-chain Data**：最基础，读链上数据必须带 chain id、block number、contract address 和读取时间，否则不同链不同时间的数据容易混在一起。
- **Contract Docs**：ABI 只有函数签名，文档补业务语义。但文档可能过期，要用链上数据交叉验证。
- **ABI / Event**：Agent 理解合约能力和历史行为的核心。能调用不等于应该调用，写交易前要过权限、余额、滑点、模拟检查。
- **Transaction History**：判断授权状态、风险交互、合约是否升级。证据必须保留 tx hash 和区块号，不能只留自然语言总结。
- **Explorer Context**：区块浏览器提供的可视化证据，给 explorer link 比给文字结论更可信。
- **Indexing Context**：把链上事件整理成产品可查询的数据。必须关注索引同步状态，落后的索引不能当事实。
- **Citation**：每条链上结论都要附引用（交易哈希、合约地址、explorer 链接等）。没有引用的链上解释只是观点，带引用的才是可验证的。

## 产品研究启发

Chain-aware Context 是所有链上 Agent 的输入层。没有这层，后面的 Web3 Tool Use、Agent Wallet、Agent Workflow 都建在不可靠的基础上。

从产品视角看，值得追问：
- 现有 AI+Web3 产品中，哪些真正接入了链上数据，哪些只是在"猜"？
- 上下文包的组装体验怎么设计？完全自动化 vs 用户手动选择，各有取舍。
- Citation 机制是否可以作为产品的差异化卖点——让用户能验证每一步操作的事实依据。
